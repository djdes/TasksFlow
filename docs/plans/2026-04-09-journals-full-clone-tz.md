# ТЗ: перенос раздела «Журналы» из lk.haccp-online.ru в Wesetup

**Дата:** 2026-04-09  
**Статус:** рабочее ТЗ для реализации 35+ журналов на своей БД

## 1. Цель
Скопировать UX/логику раздела журналов из `https://lk.haccp-online.ru/docs/1` в текущий проект (`Wesetup.ru`), сохранив:
- структуру раздела и список журналов;
- сценарии работы (создание, редактирование, просмотр, закрытие, фильтры, поиск, модальные окна, таблицы, печать/PDF где есть);
- поведение интерфейса и маршруты внутри раздела.

Источник данных и запись должны идти **только в нашу БД** (Prisma/PostgreSQL), без зависимости от backend источника.

## 2. Техническая стратегия
Не делать 35 изолированных копий. Делать `движок + конфиги`:
- `journal engine`: общий рендерер, CRUD, статусы, права, документы, периодичность, экспорт;
- `journal config`: описание каждого журнала (поля, типы, правила, период, табы, действия, валидации);
- `behavior adapters`: нестандартная логика отдельных журналов (кастомные правила/калькуляции/сетки).

## 3. Границы работ
Включено:
- весь раздел «Журналы» и все экраны после входа в конкретный журнал;
- все кнопки/модалки/таблицы/формы в рамках журналов;
- интеграция с существующими сущностями проекта (пользователи, зоны, оборудование, продукты, отчеты).

Не включено (отдельным ТЗ):
- биллинг/тарифы источника;
- внешние системы источника, если они не относятся к журналам;
- перенос прод-данных источника.

## 4. Модель данных (целевая)
Минимальный каркас:
- `JournalTemplate` (код, имя, категория, настройки периодичности, схема полей, ui-параметры);
- `JournalDocument` (журнал, период, статус, ответственный, метаданные);
- `JournalEntry` (строка/ячейка данных, дата/время, payload, аудит);
- `JournalActionLog` (кто/что/когда сделал);
- `JournalAttachment` (файлы/фото, если требуются).

Принцип: все специфичные поля журнала хранятся в schema-driven формате (JSON + валидация), а не отдельными таблицами на каждый журнал.

## 5. Реализация по этапам
1. Инвентаризация источника
- пройти все доступные журналы;
- зафиксировать маршруты, вкладки, действия, состояния, ошибки, ограничения;
- собрать матрицу `journal -> screens -> actions -> api behavior`.

2. Унификация
- выделить 4-6 типов журналов (event, document-grid, checklist, periodic-plan, tracked, custom);
- описать единый контракт конфигов и адаптеров.

3. Движок
- доработать текущие `src/components/journals/*`, `src/lib/*document*`, API `src/app/api/journal-documents/*`;
- закрыть общий CRUD, статусы, периодизацию (`15_days | month | year | custom`), фильтры, сортировки.

4. Подключение журналов пакетами
- подключать не по одному, а пакетами по 3-5 журналов;
- на пакет: конфиги + адаптеры + smoke + фиксация parity.

5. Финализация
- регресс по всем журналам;
- фиксация parity-таблицы (что 1:1, что адаптировано);
- подготовка к прод-деплою.

## 6. Критерии приемки
- в разделе `/journals` отображаются все целевые журналы;
- у каждого журнала работают ключевые сценарии: create/read/update/close/search/filter;
- все данные записываются и читаются из нашей БД;
- отсутствуют 404/500 на маршрутах журналов;
- lint/build проходят на измененном контуре;
- на каждый журнал есть smoke-чеклист.

## 7. Блокеры и риски
Текущий тест доступа (2026-04-09):
- логин успешен, вход ведет на `https://lk.haccp-online.ru/docs/1`;
- прямые URL части журналов (`daily_rejection`, `raw_storage_control`, `defrosting_control`) редиректят обратно на `/docs/1`.

Вывод: для полного 1:1 копирования всех журналов нужен аккаунт/роль с доступом ко всем внутренним страницам журналов. Без этого часть логики можно только реконструировать по косвенным данным.

## 8. Рабочий порядок дальше
1. Подтвердить аккаунт с полным доступом к 35+ журналам.
2. Сделать полный source-capture матрицы журналов.
3. Реализовать пакетами до полного покрытия.
